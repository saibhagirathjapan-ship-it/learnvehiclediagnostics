## story
fig: figures/v2-c2-f1_sid-bit6-free.svg
en: The reply's first byte is `50h`; the request's was `10h`. Line them up and `50h` is `10h` with a single bit switched on — and that is no accident.

To see which bit — and why it is always free to switch — read a **service identifier** (the first byte, the **SID**) in binary.
jp: 応答の先頭バイトは `50h`、リクエストは `10h` でした。並べてみると、`50h` は `10h` に1ビットだけ立てたもの ― これは偶然ではありません。

どのビットか、そしてなぜそれが常に立てて構わないのかを見るために、**サービス識別子**（先頭バイト、**SID**）を2進数で読みます。
--
en: It is not only `10h`. Take any request the ECU accepts — `22h`, `3Eh`, even a high one like `85h` — and one bit never changes:

**bit 6 is `0` on every request.** The standard guarantees it.
jp: `10h` だけではありません。ECUが受け付けるどのリクエストでも ― `22h`、`3Eh`、`85h` のような大きな番号でも ― 変わらないビットが1つあります：

**ビット6は、どのリクエストでも `0`。** 標準がそれを保証しています。
--
en: Bits 6 and 7 both sit at `0` on `10h` — so why claim **bit 6**, not bit 7?

Because **bit 7 is not free**: services such as ControlDTCSetting (`85h`) already set it. Bit 6 is the *one* bit that is `0` on **every** service — so that is the bit to use.
jp: `10h` ではビット6もビット7も `0` ― では、なぜ **ビット6** で、ビット7ではないのでしょう？

**ビット7は空いていない**からです：ControlDTCSetting（`85h`）のようなサービスがすでに使っています。ビット6こそ、**どの**サービスでも `0` である*唯一*のビット ― だからそれを使います。
--
fig: figures/v2-c2-f2_plus-0x40-bitflip.svg
en: Now, *why* is setting a bit the same as adding a number? Because setting a bit **adds its place value**.

In a byte, **bit 6 is the `40h` place** — on its own, the binary `0100 0000` is `40h` (64 in decimal). So "set bit 6" and "add `40h`" are exactly the same operation — on any SID.
jp: では、*なぜ*ビットを立てることが数を足すことと同じなのでしょう？　ビットを立てると、その**桁の値が足される**からです。

1バイトの中で、**ビット6は `40h` の桁** ― それだけなら、2進数 `0100 0000` は `40h`（10進で64）です。だから「ビット6を立てる」と「`40h` を足す」は、どのSIDでも、まったく同じ操作です。
--
en: So take the request `10h` and set bit 6: `10h + 40h = 50h`. That fixed step finally has its name — **+40h**.

And it is the same move for every service: `22h`→`62h`, `3Eh`→`7Eh`, `85h`→`C5h`. Read any first byte at a glance — **bit 6 clear = a request; bit 6 set = its positive reply.**
jp: そこでリクエスト `10h` のビット6を立てます：`10h + 40h = 50h`。この決まった1手に、ようやく名前がつきます ― **+40h**。

そして、どのサービスでも同じ1手です：`22h`→`62h`、`3Eh`→`7Eh`、`85h`→`C5h`。先頭バイトをひと目で読めます ― **ビット6が0ならリクエスト、ビット6が1ならその肯定応答。**
--
fig: figures/v2-c2-f3_rule-not-field.svg
en: One thing worth seeing before you move on.

`+40h` is a **rule**, not a stored value: the ECU never *chooses* the response byte, and never writes it into a "message type" field — it **computes** it from the request.
jp: 先へ進む前に、見ておく価値のあることが1つ。

`+40h` は**規則**であり、保存された値ではありません ― ECUが応答バイトを*選ぶ*ことも「メッセージ種別」フィールドに書き込むこともなく、リクエストから**計算**します。
--
en: That single choice pays off three ways:

- the tester can **predict** the exact answer byte before it arrives;
- a positive and a negative reply can **never be confused** (`50h` versus `7Fh`);
- and **no byte is wasted** carrying a type the number already encodes.
jp: この1つの選択は、3つの形で報われます：

- テスターは答えのバイトを、届く前に**予測**できる；
- 肯定応答と否定応答を**取り違えることがない**（`50h` と `7Fh`）；
- 番号がすでに符号化している種別を運ぶために、**1バイトも無駄にしない**。

## footer
:::elaborate
q_en: In your own words — why make the positive SID a fixed `+40h` rule, instead of a separate "response type" byte the ECU fills in?
q_jp: 自分の言葉で ― なぜ肯定SIDを、ECUが埋める別の「応答種別」バイトではなく、決まった `+40h` の規則にするのでしょう？
a_en: A rule needs no extra byte and can never disagree with itself: the type is *derived* from the SID, so there is nothing to fill in wrong. A separate field would cost a byte on every message and could contradict the SID — say "positive" while carrying a request number. The bit-6 rule makes the two halves of the conversation provably match.
a_jp: 規則なら余分なバイトが要らず、自分自身と矛盾しません：種別はSIDから*導かれる*ので、間違って埋める余地がありません。別フィールドなら毎メッセージ1バイトを消費し、SIDと矛盾し得ます ― リクエスト番号を運びながら「肯定」と言うなど。ビット6の規則は、会話の2つの半分が確実に一致することを保証します。
:::

ISO 14229-1:2020 | cl.8.4 | SID assignment: 01h–0Fh ISO 15031-5/SAE J1979; 10h–3Eh this document; the 8xh services (e.g. ControlDTCSetting 85h, cl.10.8) have bit 7 set — so bit 7 is not universally free.
