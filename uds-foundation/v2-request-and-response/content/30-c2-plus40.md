## story
fig: figures/v2-c2-f1_sid-bit6-free.svg
en: The reply's first byte is `50`; the request's was `10`. Line them up and `50` is `10` with a single bit switched on — and that is no accident.

To see which bit — and why it is always free to switch — read a **service identifier** (the first byte, the **SID**) in binary.
jp: 応答の先頭バイトは `50`、リクエストは `10` でした。並べてみると、`50` は `10` に1ビットだけ立てたもの ― これは偶然ではありません。

どのビットか、そしてなぜそれが常に立てて構わないのかを見るために、**サービス識別子**（先頭バイト、**SID**）を2進数で読みます。
--
en: It is not only `10`. Take any request the ECU accepts — `22`, `3E`, even a high one like `85` — and one bit never changes:

**bit 6 is `0` on every request.** The standard guarantees it.
jp: `10` だけではありません。ECUが受け付けるどのリクエストでも ― `22`、`3E`、`85` のような大きな番号でも ― 変わらないビットが1つあります：

**ビット6は、どのリクエストでも `0`。** 標準がそれを保証しています。
--
en: Bits 6 and 7 both sit at `0` on `10` — so why claim **bit 6**, not bit 7?

Because **bit 7 is not free**: services such as ControlDTCSetting (`85`) already set it. Bit 6 is the *one* bit that is `0` on **every** service — so that is the bit to use.
jp: `10` ではビット6もビット7も `0` ― では、なぜ **ビット6** で、ビット7ではないのでしょう？

**ビット7は空いていない**からです：ControlDTCSetting（`85`）のようなサービスがすでに使っています。ビット6こそ、**どの**サービスでも `0` である*唯一*のビット ― だからそれを使います。
--
fig: figures/v2-c2-f2_plus-0x40-bitflip.svg
en: So set bit 6, and `10` becomes `50`. That one fixed step has a name: **+0x40** — you add hex `40`.

And it is the same move for every service: `22`→`62`, `3E`→`7E`, `85`→`C5`. Read any first byte at a glance — **bit 6 clear = a request; bit 6 set = its positive reply.**
jp: そこでビット6を立てると、`10` は `50` になります。この決まった1手には名前があります ― **+0x40**（16進の `40` を足す）。

そして、どのサービスでも同じ1手です：`22`→`62`、`3E`→`7E`、`85`→`C5`。先頭バイトをひと目で読めます ― **ビット6が0ならリクエスト、ビット6が1ならその肯定応答。**
--
fig: figures/v2-c2-f3_rule-not-field.svg
en: One thing worth seeing before you move on.

`+0x40` is a **rule**, not a stored value: the ECU never *chooses* the response byte, and never writes it into a "message type" field — it **computes** it from the request.
jp: 先へ進む前に、見ておく価値のあることが1つ。

`+0x40` は**規則**であり、保存された値ではありません ― ECUが応答バイトを*選ぶ*ことも「メッセージ種別」フィールドに書き込むこともなく、リクエストから**計算**します。
--
en: That single choice pays off three ways:

- the tester can **predict** the exact answer byte before it arrives;
- a positive and a negative reply can **never be confused** (`50` versus `7F`);
- and **no byte is wasted** carrying a type the number already encodes.
jp: この1つの選択は、3つの形で報われます：

- テスターは答えのバイトを、届く前に**予測**できる；
- 肯定応答と否定応答を**取り違えることがない**（`50` と `7F`）；
- 番号がすでに符号化している種別を運ぶために、**1バイトも無駄にしない**。

## footer
:::elaborate
q_en: In your own words — why make the positive SID a fixed `+0x40` rule, instead of a separate "response type" byte the ECU fills in?
q_jp: 自分の言葉で ― なぜ肯定SIDを、ECUが埋める別の「応答種別」バイトではなく、決まった `+0x40` の規則にするのでしょう？
a_en: A rule needs no extra byte and can never disagree with itself: the type is *derived* from the SID, so there is nothing to fill in wrong. A separate field would cost a byte on every message and could contradict the SID — say "positive" while carrying a request number. The bit-6 rule makes the two halves of the conversation provably match.
a_jp: 規則なら余分なバイトが要らず、自分自身と矛盾しません：種別はSIDから*導かれる*ので、間違って埋める余地がありません。別フィールドなら毎メッセージ1バイトを消費し、SIDと矛盾し得ます ― リクエスト番号を運びながら「肯定」と言うなど。ビット6の規則は、会話の2つの半分が確実に一致することを保証します。
:::

:::reading
ISO 14229-1:2020 | cl.9.3, NOTE | "bit 6 of the SI byte value indicat[es] the service type. All request messages have SI bit 6 = 0. All positive response messages have SI bit 6 = 1" — except the periodic data responses of ReadDataByPeriodicIdentifier ($2A).
ISO 14229-1:2020 | cl.8.4 | SID assignment: 01–0F ISO 15031-5/SAE J1979; 10–3E this document; the 8x services (e.g. ControlDTCSetting 85, cl.10.8) have bit 7 set — so bit 7 is not universally free.
:::
