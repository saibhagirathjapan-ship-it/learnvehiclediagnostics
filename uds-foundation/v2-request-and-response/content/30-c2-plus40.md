## bar
:::panel src=figures/v2-c2-f1_plus-0x40-bitflip.svg
en: The reply's first byte is `50`. The request's was `10`. That is no coincidence — so ask: how could you turn *any* request into "the answer to it" without spending a single extra byte?
jp: 応答の先頭バイトは `50`。リクエストは `10` でした。偶然ではありません ― では問いましょう：*どんな*リクエストも、余分なバイトを1つも使わずに「それへの答え」に変えるには、どうすればいい？
--
en: Look at the service number in bits. It only ever uses the low bits — **bit 6 just sits there, unused. Free.**
jp: サービス番号をビットで見てください。使うのは下位ビットだけ ― **ビット6はただそこにあり、使われていません。空いています。**
--
en: So use it. Set bit 6, and `10` becomes `50`. That one fixed step has a name: **+0x40**. The answer is now unmistakable, and it cost nothing.
jp: なら使いましょう。ビット6を立てると、`10` は `50` になります。この決まった1手には名前があります：**+0x40**。答えは一目瞭然になり、しかもコストはゼロ。
--
en: And it is the same move across the services in this course: `22` becomes `62`, `3E` becomes `7E`. So you can read a first byte at a glance: bit 6 clear = a request, bit 6 set = its positive reply.
jp: そして、この講座で扱うサービスすべてで同じ1手です：`22` は `62` に、`3E` は `7E` に。だから先頭バイトをひと目で読めます：ビット6が0＝リクエスト、ビット6が1＝それへの肯定応答。
:::

## leg:examples
:::en
Because `+0x40` is a plain rule on one bit, the request and its positive response sit exactly `0x40` apart. You never memorise pairs — you compute them. (One specialised service — reading *periodic* data, `$2A` — is the standard's lone exception; every ordinary request/response service follows `+0x40`.)
:::
:::jp
`+0x40` は1ビットに対する単純な規則なので、リクエストとその肯定応答は、ちょうど `0x40` だけ離れています。ペアを暗記する必要はありません ― 計算するのです。（1つの特殊なサービス ― *周期*データの読み取り `$2A` ― だけが標準上の例外で、通常のリクエスト／応答サービスはすべて `+0x40` に従います。）
:::

:::figure src=figures/v2-c2-f2_plus-0x40-examples.svg
en: **One rule, every service.** Add `0x40` to the request SID and you have the positive-response SID. The gap is always the same bit.
jp: **1つの規則、すべてのサービス。** リクエストSIDに `0x40` を足せば、肯定応答SIDになります。差は常に同じ1ビット。
:::

## leg:constant
:::en
Setting bit 6 is a **rule**, not a stored value. The ECU does not choose the response byte or write it into a "message type" field — it *computes* it from the request. That has real consequences: the tester can predict the exact answer byte before it arrives, a positive and negative reply can never be confused (`50` vs `7F`), and no byte is wasted carrying a type the number already encodes.
:::
:::jp
ビット6を立てるのは**規則**であり、保存された値ではありません。ECUが応答バイトを選んだり「メッセージ種別」フィールドに書き込んだりするのではなく、リクエストから*計算*します。これには実際の効果があります：テスターは答えのバイトを届く前に予測でき、肯定応答と否定応答を取り違えることがなく（`50` と `7F`）、番号がすでに符号化している種別を運ぶために1バイトも無駄にしません。
:::

:::elaborate
q_en: In your own words — why make the positive SID a fixed `+0x40` rule instead of a separate "response type" byte the ECU fills in?
q_jp: 自分の言葉で ― なぜ肯定SIDを、ECUが埋める別の「応答種別」バイトではなく、決まった `+0x40` の規則にするのでしょう？
a_en: A rule needs no extra byte and can never disagree with itself: the type is *derived* from the SID, so there is nothing to fill in wrong. A separate field would cost a byte on every message and could contradict the SID (say "positive" while carrying a request number). The bit-6 rule makes the two halves of the conversation provably match.
a_jp: 規則なら余分なバイトが要らず、自分自身と矛盾しません：種別はSIDから*導かれる*ので、間違って埋める余地がありません。別フィールドなら毎メッセージ1バイトを消費し、SIDと矛盾し得ます（リクエスト番号を運びながら「肯定」と言うなど）。ビット6の規則は、会話の2つの半分が確実に一致することを保証します。
:::
