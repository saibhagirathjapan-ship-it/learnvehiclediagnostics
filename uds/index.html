SECTION 00

Why diagnostics existなぜ「診断」が必要なのか

A modern car is a network of computers. When something is wrong, a human needs a way to ask each computer what it knows. That conversation is what diagnostics is.

現代の車は、たくさんのコンピューターのネットワークです。何か不具合があったとき、人間はそれぞれのコンピューターに「今どうなっている？」と尋ねる手段が必要です。その会話こそが「診断（diagnostics）」です。

Start here · まずここから

The dashboard warning light turns on. It tells you something is wrong — but not what, not where, not why. A mechanic cannot open a chip and look inside. So how do they find the fault?

メーターの警告灯が点きました。これは「何かおかしい」と教えてくれますが、何が・どこで・なぜまでは分かりません。整備士はチップを開けて中を覗くことはできません。では、どうやって故障を見つけるのでしょう？

C1

The workshop scene: two parties, one cable

整備現場の風景 ― 二者と、一本のケーブル

In the workshop, a diagnostic tester (a laptop or a handheld tool) is plugged into the car. It connects through the OBD port — the standard socket under the dashboard. Through that one cable, the tester can reach the car's computers.

整備工場では、診断ツール（テスター）― ノートPCや手持ちの専用機 ― を車につなぎます。接続口はOBDポート、ダッシュボード下にある共通のコネクタです。この一本のケーブルを通して、テスターは車の中のコンピューターへ届きます。

Each of those computers is called an ECU (Electronic Control Unit) — a small computer that controls one job: the engine, the brakes, the door locks, the airbags. A car can have 20 to 100 of them.

その一つひとつのコンピューターをECU（電子制御ユニット）と呼びます。エンジン、ブレーキ、ドアロック、エアバッグ ― それぞれ一つの役割を受け持つ小さなコンピューターです。1台の車に20〜100個ほど載っています。

C2

What UDS is, in one breath

UDSとは何か ― ひと息で

UDS stands for Unified Diagnostic Services. It is the shared language the tester and the ECU both agree to speak. Without a shared language, every carmaker's tool would need a different dictionary for every ECU. UDS is that one dictionary.

UDSは Unified Diagnostic Services（統一診断サービス）の略です。テスターとECUが「この言葉で話そう」と互いに合意した共通の言語です。共通言語が無ければ、メーカーやECUごとに別々の辞書が必要になってしまいます。UDSは、その「一つの辞書」です。

UDS is written down as an international standard, ISO 14229. It defines a set of services — each service is one thing the tester can ask an ECU to do: read a fault, reset itself, unlock a protected function, reprogram its software. This page will teach two of them: $10 and $11.

UDSは国際規格 ISO 14229 として文書化されています。UDSは複数のサービス（services）を定義します。各サービスは「テスターがECUに頼める一つの動作」です ― 故障を読む、自分を再起動する、保護された機能を解除する、ソフトを書き換える、など。このページでは、そのうち $10 と $11 の2つを学びます。

Key idea · 要点 $ means "hexadecimal." $10 is just the number 16 written in base-16. Every UDS service has such a number — its Service Identifier (SID). We will meet the SID properly in C8. $ は「16進数」という意味です。$10 は10進数の 16 を16進で書いたものにすぎません。UDSの各サービスにはこのような番号 ― サービス識別子（SID）があります。SIDはC8で正式に扱います。
Section recap · まとめ

Diagnostics is a conversation between a tester and the car's ECUs. UDS (ISO 14229) is the shared language for that conversation, made of numbered services. Next: who speaks first, and how one exchange is shaped.

診断とは、テスターと車のECUの会話です。UDS（ISO 14229）はその会話のための共通言語で、番号のついたサービスでできています。次は、どちらが先に話すのか、そして1回のやり取りがどんな形をしているのかを見ます。

SECTION 01

The big picture: where UDS lives全体像 ― UDSはどこに住んでいるのか

Before any hex bytes, build the mental model: who is the client, who is the server, what one round-trip looks like, where UDS sits inside the ECU, and how it maps onto the OSI layers.

16進のバイトに入る前に、頭の中の地図を作ります ― 誰がクライアントで、誰がサーバーか。1往復はどんな形か。UDSはECUの中のどこに座り、OSI階層のどこに当たるのか。

C3

Client and server: the tester asks, the ECU answers

クライアントとサーバー ― テスターが尋ね、ECUが答える

UDS follows a strict client–server pattern. The tester is the client: it always speaks first. The ECU is the server: it waits, and only replies when asked. The ECU never starts a UDS conversation on its own.

UDSは厳格なクライアント・サーバー方式です。テスターがクライアントで、いつも先に話します。ECUがサーバーで、じっと待ち、尋ねられたときだけ答えます。ECUの側から会話を始めることはありません。

This is the same idea as a web browser and a web server. You (the browser) type a request; the server sends a page back. In UDS, the tester sends a request; the ECU sends a response back.

これはWebブラウザとWebサーバーと同じ考え方です。あなた（ブラウザ）がリクエストを送ると、サーバーがページを返します。UDSでは、テスターがリクエストを送り、ECUがレスポンスを返します。

Watch out · 注意 "Server" here does not mean a big machine in a data center. In UDS, server = the ECU being asked. A tiny brake controller is a "server." The role is about who answers requests, not about size. ここでの「サーバー」は、データセンターの大きな機械のことではありません。UDSではサーバー＝尋ねられる側のECUです。小さなブレーキ制御ユニットも「サーバー」です。役割はリクエストに答える側は誰かという意味で、大きさとは無関係です。
C4

One exchange: request in, response out

1回のやり取り ― リクエストを送り、レスポンスを受ける

The whole of UDS is built from one simple beat, repeated: the tester sends one request, then waits for one response. It does not send the next request until the current one is answered (or a timer runs out). This is called a request–response exchange.

UDS全体は、たった一つの拍子（ビート）の繰り返しでできています ― テスターが1つのリクエストを送り、1つのレスポンスを待ちます。今のやり取りに答えが返る（またはタイマーが切れる）まで、次のリクエストは送りません。これをリクエスト・レスポンスのやり取りと呼びます。

The answer can be one of two kinds: a positive response (“done, here is the result”) or a negative response (“I cannot — here is why”). We meet both in Section 2.

答えには2種類あります ― 肯定応答（positive response）（「できました。結果はこれです」）か、否定応答（negative response）（「できません。理由はこれです」）です。両方ともセクション2で扱います。

C5

Inside the ECU: the diagnostic stack (DCM & DM)

ECUの内部 ― 診断スタック（DCMとDM）

Where does UDS actually run inside the ECU? An ECU runs its main job (say, controlling the engine) in application software. Alongside it sits a dedicated block that handles diagnostics. In AUTOSAR — the common software framework for ECUs — this block is the DCM: Diagnostic Communication Manager.

UDSは、ECUの中で実際にはどこで動くのでしょう。ECUは本来の仕事（たとえばエンジン制御）をアプリケーションソフトで行います。その隣に、診断を専門に受け持つブロックがあります。ECU向けの共通ソフト基盤である AUTOSAR では、このブロックが DCM（Diagnostic Communication Manager／診断通信マネージャ）です。

The DCM is the part that understands UDS. When a request arrives, the DCM reads the service number, checks the rules (right session? unlocked?), asks the application for the data, and builds the response. A second block, the DEM: Diagnostic Event Manager, is the ECU's fault memory — it stores the trouble codes (DTCs) that services like $19 later read out.

DCMはUDSを理解する担当です。リクエストが届くと、DCMはサービス番号を読み、ルール（正しいセッションか？ ロックは解除済みか？）を確認し、必要なデータをアプリケーションに問い合わせ、レスポンスを組み立てます。もう一つのブロック DEM（Diagnostic Event Manager／診断イベントマネージャ）はECUの故障メモリで、$19 などのサービスが後で読み出す故障コード（DTC）を保管します。

The word "DM" · 「DM」という言葉 You will hear "DM" used loosely for the diagnostic manager. In AUTOSAR terms this splits into two: DCM (handles the UDS conversation) and DEM (holds the fault memory). Together they are the ECU's diagnostic stack. 現場では診断マネージャを大まかに「DM」と呼ぶことがあります。AUTOSARではこれが2つに分かれます ― DCM（UDSの会話を処理）と DEM（故障メモリを保持）。この2つを合わせたものがECUの診断スタックです。
C6

The OSI map: UDS is the top, CAN is the bottom

OSIの地図 ― UDSは一番上、CANは一番下

Engineers describe any communication as a stack of layers — the OSI model has seven. The top layers care about meaning (“reset the ECU”); the bottom layers care about wires and voltages. UDS is not the whole stack — it is only the top.

エンジニアは通信を階層（レイヤー）の積み重ねとして説明します ― OSIモデルは7階層です。上の層は意味（「ECUを再起動せよ」）を扱い、下の層は電線や電圧を扱います。UDSはスタック全体ではなく、その一番上だけです。

  • UDS (ISO 14229) defines the meaning of messages — the application layer (OSI layer 7, plus session behaviour). It says nothing about wires.
  • UDS（ISO 14229）はメッセージの意味を定義します ― アプリケーション層（OSI第7層＋セッションの振る舞い）。配線については何も言いません。
  • DoCAN (ISO 15765-2) is the transport. A UDS message may be longer than one CAN frame, so this layer chops it up, sends the pieces, and reassembles them (OSI layers 3–4).
  • DoCAN（ISO 15765-2）はトランスポートです。UDSメッセージは1つのCANフレームより長いことがあるため、この層が分割・送信・再結合します（OSI第3〜4層）。
  • CAN (ISO 11898) is the physical bus — the actual wires and electrical signals (OSI layers 1–2).
  • CAN（ISO 11898）は物理バスです ― 実際の配線と電気信号（OSI第1〜2層）。
Why this matters · なぜ重要か Because UDS only defines meaning, the same UDS services can ride over CAN today, or over Ethernet (DoIP) tomorrow. Swap the lower layers; the language on top stays the same. Remember this — it is exactly the seam SOVD later widens. UDSは意味だけを定義するので、同じUDSサービスを、今日はCANの上で、明日はEthernet（DoIP）の上で走らせられます。下層を差し替えても、上の言語は変わりません。これを覚えておいてください ― 後でSOVDが押し広げるのは、まさにこの継ぎ目です。
C7

1つのリクエストが処理される流れ ― 端から端まで

Now put C5 and C6 together. Follow a single request from the tester's screen to the ECU's answer. Each step is one layer doing its one job.

ここでC5とC6を合わせます。テスターの画面からECUの答えまで、1つのリクエストを追いかけましょう。各ステップは、ある層が自分の一つの仕事をしている場面です。

Section recap · まとめ

The tester is the client, the ECU is the server, and they trade one request for one response. Inside the ECU, the DCM understands UDS while the DEM holds faults. UDS is only the top of the OSI stack; DoCAN and CAN carry it. Next: the exact shape of the bytes.

テスターがクライアント、ECUがサーバーで、1つのリクエストと1つのレスポンスを交換します。ECU内部ではDCMがUDSを理解し、DEMが故障を保持します。UDSはOSIスタックの一番上だけで、DoCANとCANがそれを運びます。次は、バイトの正確な形です。

SECTION 02

Anatomy of a UDS messageUDSメッセージの解剖

A UDS message is just a short list of bytes. Learn to read those bytes and you can read any UDS exchange. There are only four shapes to know: the request, the positive response, the negative response, and the sub-function.

UDSメッセージは、短いバイトの並びにすぎません。このバイトを読めるようになれば、どんなUDSのやり取りも読めます。覚える形はたった4つ ― リクエスト、肯定応答、否定応答、そしてサブファンクションです。

C8

The first byte: the Service Identifier (SID)

最初のバイト ― サービス識別子（SID）

Every UDS request begins with one byte that names the service. This byte is the SID — Service Identifier. It answers the question “which service am I asking for?” Everything after it is the detail.

すべてのUDSリクエストは、サービスの名前を表す1バイトで始まります。このバイトが SID（Service Identifier／サービス識別子）です。「どのサービスを頼むのか？」に答えるバイトで、その後ろに続くのは細かい指定です。

The two services on this page have these SIDs: $10 = Diagnostic Session Control, $11 = ECU Reset. Read $10 as “ten in hex” = the number 16. You do not need to memorise the number — you read the name from a table.

このページの2つのサービスのSIDは、$10 ＝ 診断セッション制御、$11 ＝ ECUリセットです。$10 は「16進の10」＝10進の16です。数値を暗記する必要はなく、表から名前を引けば十分です。

C9

The request shape: SID + parameters

リクエストの形 ― SID＋パラメータ

A request is the SID followed by zero or more parameter bytes. Some services need a sub-function byte first (a menu choice within the service); some need data. The rule of the shape is fixed per service and written in ISO 14229.

リクエストは、SIDのあとに0個以上のパラメータバイトが続きます。サービスによっては最初にサブファンクションバイト（そのサービス内の「メニュー選択」）が必要で、データが必要なものもあります。形のルールはサービスごとに固定で、ISO 14229に書かれています。

Here is a complete $11 request — just two bytes: “ECU Reset” + “do a hard reset.” Short messages like this fit in a single CAN frame, so DoCAN has nothing to split.

下は完全な $11 リクエストです。わずか2バイト ―「ECUリセット」＋「ハードリセットして」。この程度の短いメッセージは1つのCANフレームに収まるので、DoCANが分割する必要はありません。

C10

The positive response: SID + 0x40

肯定応答 ― SID＋0x40

When the ECU succeeds, it replies with a positive response. The trick is simple and always the same: the first byte is the request SID plus 0x40. So $10 comes back as $50, and $11 comes back as $51. That +0x40 is the ECU saying “yes, that one — done.”

ECUが成功すると、肯定応答を返します。コツは簡単で、いつも同じです ― 最初のバイトはリクエストのSIDに 0x40 を足した値です。だから $10 は $50 として、$11 は $51 として返ってきます。この +0x40 が、ECUの「はい、それですね ― 完了しました」という合図です。

C11

The negative response: 0x7F + SID + reason

否定応答 ― 0x7F＋SID＋理由

When the ECU cannot do what was asked, it does not stay silent — it sends a negative response, always three bytes: 7F then the SID it is rejecting then a reason code. The reason code is the NRC — Negative Response Code. It turns “no” into “no, because…”

頼まれたことができないとき、ECUは黙り込みません ― 否定応答を返します。いつも3バイトで、7F、次に拒否したSID、次に理由コードです。理由コードが NRC（Negative Response Code／否定応答コード）です。これが「いいえ」を「いいえ、なぜなら…」に変えます。

Some NRCs you will meet again in Sections 4–5:

セクション4〜5で再会するNRCの例です：

NRCNamePlain meaning · やさしい意味

13incorrectMessageLength…"Your message is the wrong length." / メッセージの長さが違う。
22conditionsNotCorrect"Not right now — conditions aren't met." / 今は条件が整っていない。

Don't confuse them · 混同注意 7F appears in two roles: as the first byte it marks "this is a negative response"; as an NRC value it means "service not supported in active session." Position tells you which — the first byte is never an NRC, and the NRC is always the last byte. 7F は2つの役割で現れます ― 先頭バイトなら「これは否定応答」の印、NRC値なら「このセッションでサービス不可」。位置で見分けます ― 先頭バイトがNRCになることはなく、NRCは常に最後のバイトです。
C12

Sub-functions, and the “stay quiet” bit

サブファンクションと「黙っていて」ビット

Many services carry a sub-function byte right after the SID — a menu choice inside the service. For $10, the sub-function picks which session; for $11, it picks which kind of reset.

多くのサービスは、SIDの直後にサブファンクションバイトを持ちます ― サービス内の「メニュー選択」です。$10 ではサブファンクションがどのセッションかを、$11 ではどの種類のリセットかを選びます。

The top bit of the sub-function byte (bit 7, value 0x80) is special: it is the suppressPositiveResponse bit. Set it to say “do it, but don't bother replying if it works.” The ECU still replies if it fails (you always hear about problems).

サブファンクションバイトの最上位ビット（ビット7、値 0x80）は特別です ― suppressPositiveResponse（肯定応答抑制）ビットです。これを立てると「実行して。ただし成功したなら返事はいらない」という意味になります。失敗したときはECUは必ず返事をします（問題は必ず知らされます）。

C13

The service catalog: six families

サービスカタログ ― 6つのファミリー

ISO 14229 groups its services into six functional families. You don't need them all today — but seeing the map tells you where $10 and $11 sit and what a tester can do beyond them. Both of our services live in the first family: Diagnostic & Communication Management.

ISO 14229は、サービスを6つの機能ファミリーに分類します。今日すべてを覚える必要はありませんが、地図を見れば $10 と $11 がどこに座っているか、そしてテスターが他に何をできるかが分かります。私たちの2つは最初のファミリー ― 診断・通信管理に属します。

Read the bytes and you read UDS. Request = SID + parameters. Success = SID + 0x40. Failure = 7F + SID + NRC. A sub-function is a menu choice, and its top bit can silence a positive reply. Next: the sessions those bytes switch between.

バイトが読めればUDSが読めます。リクエスト＝SID＋パラメータ。成功＝SID＋0x40。失敗＝7F＋SID＋NRC。サブファンクションはメニュー選択で、その最上位ビットで肯定応答を黙らせられます。次は、それらのバイトが切り替える「セッション」です。

SECTION 03

Sessions: the ECU as a state machineセッション ― 状態機械としてのECU

An ECU is not always in the same mood. It has states, called sessions, and each session allows a different set of actions. Understand sessions and $10 stops being mysterious — because $10 is the service that changes the state.

ECUはいつも同じ「気分」ではありません。状態を持ち、それをセッションと呼びます。セッションごとに許される操作が違います。セッションが分かれば $10 は謎でなくなります ― $10 こそ、その状態を切り替えるサービスだからです。

C14

The ECU has states, not just one mode

ECUは複数の状態を持つ

Think of a phone. Locked, it only shows the time and takes emergency calls. Unlocked, it does everything. Same phone — different state, different permissions. An ECU works the same way. Its states are called diagnostic sessions.

スマホを思い浮かべてください。ロック中は時刻の表示と緊急通報だけ。解除すれば何でもできます。同じスマホでも、状態が違えば許可も違います。ECUも同じです。その状態を診断セッションと呼びます。

On power-up, every ECU starts in the Default Session. It is doing its real job (driving the car) and allows only harmless diagnostics. To do more — reprogram, run tests, unlock protected data — the tester must first move the ECU into another session with $10.

電源投入時、どのECUもデフォルトセッションで始まります。本来の仕事（車を走らせる）をしていて、無害な診断だけを許します。それ以上のこと ― 書き換え、テスト実行、保護データの解錠 ― を行うには、テスターがまず $10 でECUを別のセッションへ移す必要があります。

C15

What a session unlocks

セッションが解放するもの

Sessions exist for safety. You would not want a workshop tool to reprogram the airbag controller while the car is moving. So risky services are only allowed in the right session — and often only after a security unlock ($27). A session is the coarse gate; security is the fine lock behind it.

セッションは安全のために存在します。走行中に整備ツールがエアバッグ制御を書き換えられては困ります。だから危険なサービスは、正しいセッションでのみ ― しばしばセキュリティ解錠（$27）の後にのみ ― 許されます。セッションは大まかな「門」、セキュリティはその奥の細かい「錠」です。

Session · セッションRoughly allows · おおよそ許すこと

Extended (03)

Why $11 needs this · なぜ$11に関係するか An ECU Reset ($11) is often only offered in a non-default session — you reset after reprogramming or testing. Ask for it in the wrong session and you get a negative response (7F 11 7F). Sessions are the reason a valid service can still be refused. ECUリセット（$11）は、非デフォルトのセッションでのみ提供されることがよくあります ― 書換えやテストの後にリセットするからです。違うセッションで頼むと否定応答（7F 11 7F）になります。正しいサービスでも拒否されうる理由が、このセッションです。
C16

Timing: P2, S3, and keeping the session alive

タイミング ― P2、S3、そしてセッションの維持

Two timers matter now. P2 is how long the tester waits for a response after a request. S3 is how long the ECU keeps a non-default session alive with no traffic — typically 5 seconds. If the tester goes quiet for longer than S3, the ECU gives up and drops back to the Default Session on its own.

今は2つのタイマーが重要です。P2は、リクエスト後にテスターがレスポンスを待つ時間。S3は、通信が無いときにECUが非デフォルトのセッションを保つ時間で、ふつう5秒です。テスターがS3より長く黙ると、ECUはあきらめ、自分からデフォルトセッションへ戻ります。

To stop that, the tester sends a tiny “I'm still here” message on a timer: the TesterPresent service, $3E. It carries no real work — its only job is to reset the S3 clock so the session stays open during a long operation.

それを防ぐため、テスターは一定間隔で小さな「まだここにいます」というメッセージを送ります ― TesterPresentサービス、$3E です。実際の作業はせず、S3の時計をリセットしてセッションを開き続けることだけが役割です。

Section recap · まとめ

An ECU lives in sessions — Default on power-up, then Extended or Programming for real work. Sessions gate what is allowed, for safety. S3 drops an idle session back to Default; TesterPresent ($3E) holds it open. You now have everything needed to read the first real service. Next: $10, byte by byte.

ECUはセッションの中で生きます ― 起動時はデフォルト、実作業では拡張やプログラミング。セッションは安全のために許可を制限します。S3は放置されたセッションをデフォルトへ戻し、TesterPresent（$3E）が開いたまま保ちます。これで最初の本物のサービスを読む準備が整いました。次は $10 をバイト単位で。

SERVICE $10 · SID 0x10

Diagnostic Session Control診断セッション制御（$10）

Our first real service. $10 is how the tester moves the ECU from one session to another — the “change the mood” command. Everything from Section 3 now pays off, byte by byte.

最初の本物のサービスです。$10 は、テスターがECUをあるセッションから別のセッションへ移す方法 ―「気分を変える」コマンドです。セクション3で学んだことが、ここでバイト単位で活きてきます。

C17

What $10 is for, and when you send it

$10は何のため、いつ送るのか
SID 0x10

Diagnostic Session Control changes which diagnostic session the ECU is in. It is almost always the first real request in a workshop procedure, because most useful services are locked behind a non-default session. Want to run an actuator test? Switch to Extended first. Want to reflash? Switch to Programming first.

診断セッション制御は、ECUがどの診断セッションにいるかを変えます。整備手順で最初に送る本物のリクエストになることがほとんどです。便利なサービスの多くが、非デフォルトのセッションの奥に施錠されているからです。アクチュエータ試験をしたい？ まず拡張へ。リフラッシュしたい？ まずプログラミングへ。

One-line job · ひとことで $10 answers: “Which mode should the ECU be in for what I'm about to do?” It sets the stage; the real work follows in later requests. $10 が答えるのは「これからやることのために、ECUはどのモードにいるべきか？」です。舞台を整える役で、実作業は後続のリクエストで行います。

リクエストをバイト単位で

$10 リクエストはちょうど2バイトです ― SID 10、そして欲しいセッションを表す1バイトのサブファンクション。

C19

The positive response — and the timing it carries

肯定応答 ― そこに載るタイミング情報
0x50

A positive answer to $10 starts with 50 (that's 10 + 0x40), echoes the session you asked for, and then — this is the useful part — reports the ECU's timing limits for this session: P2 and P2*, four bytes total.

$10 への肯定応答は 50（＝10 + 0x40）で始まり、頼んだセッションをそのまま返し、そして ― ここが便利な点 ― このセッションでのECUのタイミング上限、すなわち P2 と P2* を計4バイトで報告します。

P2 is the normal deadline: the ECU promises to answer any request within this time (here 50 ms). P2* (“P2 star”) is the extended deadline it may use when it first replies with a “please wait” message (NRC 78, responsePending) — up to 5000 ms here. The tester reads these numbers and sets its own timers to match.

P2 は通常の締め切りで、ECUはどのリクエストにもこの時間内（ここでは50ms）に答えると約束します。P2*（「P2スター」）は延長の締め切りで、ECUがまず「お待ちください」応答（NRC 78、responsePending）を返したときに使えます ― ここでは最大5000ms。テスターはこの数値を読み、自分のタイマーを合わせます。

Careful · 注意 Bytes are hexadecimal. 00 32 is not “32” — it is 0x0032 = 50 decimal. 01 F4 = 0x01F4 = 500 decimal, and P2* counts in 10 ms steps, so 500 × 10 = 5000 ms. Always convert before you read a time. バイトは16進です。00 32 は「32」ではなく 0x0032＝10進の50。01 F4＝0x01F4＝10進の500で、P2*は10ms刻みなので 500×10＝5000ms。時間として読む前に必ず換算します。
C20

The three sessions, side by side

3つのセッションを横並びで
SID 0x10

Same service, three destinations. Each is one sub-function, and each replies with 50 + its own echo. Here is the whole switch in one view.

同じサービスで、行き先は3つ。それぞれ1つのサブファンクションで、応答は 50 ＋自分のエコーです。切替えの全体を一枚で。

C21

実例 ― 成功、そして拒否
SID 0x10

Put it together. The tester asks for the Extended session and succeeds. Later it asks for the Programming session while the engine is running — and the ECU refuses, politely, with a reason.

まとめて見ましょう。テスターが拡張セッションを要求して成功します。その後、エンジンがかかったままプログラミングセッションを要求し ― ECUは理由をつけて丁寧に拒否します。

Service recap · $10まとめ

$10 = two bytes in (10 + session), and on success six bytes back (50 + echo + P2 + P2*). It sets the ECU's state so later services are allowed, and its response hands the tester the timing contract. Refusals come back as 7F 10 <NRC>. Next: the service that ends a procedure — $11, ECU Reset.

$10 ＝ 2バイト送信（10＋セッション）、成功時は 6バイト応答（50＋エコー＋P2＋P2*）。ECUの状態を整えて後続サービスを許可し、応答でタイミングの約束を渡します。拒否は 7F 10 <NRC> で返ります。次は手順を締めくくるサービス ― $11、ECUリセットです。

SERVICE $11 · SID 0x11

ECU ResetECUリセット（$11）

The second real service. $11 tells an ECU to restart itself — to reboot. It looks simple, and the request is simple, but which kind of restart, and what happens after, are where the care is.

2つ目の本物のサービスです。$11 はECUに自分を再起動 ― リブート ― するよう命じます。単純に見え、リクエストも単純ですが、どの種類の再起動か、そしてその後どうなるかにこそ注意が必要です。

C22

What $11 is for, and when you send it

$11は何のため、いつ送るのか
SID 0x11

ECU Reset makes the ECU restart. You send it to apply a change and to clean up. After reprogramming new software, the ECU must reboot to run it. After writing new configuration, a reset makes the settings take effect. And when an ECU is stuck, a reset is the recovery of last resort — the automotive “turn it off and on again.”

ECUリセットはECUを再起動させます。変更を反映し、後始末をするために送ります。新しいソフトを書き込んだ後は、それを動かすために再起動が必要です。新しい設定を書き込んだ後は、リセットで設定が有効になります。そしてECUがハングしたときは、リセットが最後の復旧手段 ― 自動車版の「電源を入れ直す」です。

Real-world weight · 現場での重み A reset is not free. The ECU stops doing its job for a moment while it reboots. Resetting a brake or steering controller on a moving car would be dangerous — which is exactly why the ECU may refuse the request unless conditions are safe. リセットはタダではありません。再起動の間、ECUは一瞬だけ本来の仕事を止めます。走行中の車でブレーキや操舵の制御ユニットをリセットすれば危険です ― だからこそ、条件が安全でなければECUはリクエストを拒否することがあります。
C23

The request, byte by byte

リクエストをバイト単位で
SID 0x11

Just like $10, a $11 request is two bytes: the SID 11, then a sub-function that names the kind of reset. The shape is familiar — only the meaning of the second byte changes.

$10 と同じく、$11 リクエストも2バイトです ― SID 11、そしてリセットの種類を表すサブファンクション。形はおなじみで、2バイト目の意味だけが変わります。

C24

The reset types — not all reboots are equal

リセットの種類 ― 再起動は同じではない
SID 0x11

The sub-function picks how deep the restart goes. The two you meet most are hard reset (a full reboot, like disconnecting and reconnecting power) and soft reset (restart the software, keep the hardware powered).

サブファンクションは、再起動の「深さ」を選びます。よく出会う2つは、ハードリセット（電源を抜き差しするような完全な再起動）と、ソフトリセット（ハードは通電したまま、ソフトだけ再起動）です。

肯定応答 ― その後に起きること
0x51

A positive answer is two bytes: 51 (that's 11 + 0x40) and an echo of the reset type. The ECU sends this first, and only then reboots — otherwise the tester would never hear the confirmation.

肯定応答は2バイトです ― 51（＝11 + 0x40）とリセット種別のエコー。ECUはこれを先に送り、その後で再起動します ― そうしないとテスターは確認を受け取れないからです。

The after-effect you must remember · 覚えるべき副作用 After the reboot, the ECU comes back up in the Default Session — every non-default session and every security unlock is gone. So a common pattern is: switch session, do the work, reset — and know that the reset has closed the session for you. If you need to keep working, you must $10 back into the session again. 再起動の後、ECUはデフォルトセッションで立ち上がります ― 非デフォルトのセッションもセキュリティ解錠もすべて消えます。だからよくある流れは「セッション切替 → 作業 → リセット」で、リセットがセッションを閉じてくれることを理解しておきます。続けて作業するなら、再び $10 でセッションに入り直す必要があります。
C26

Worked example: reset accepted, reset refused

実例 ― 受理されるリセット、拒否されるリセット
SID 0x11

In the Extended session, a hard reset is accepted and the ECU reboots into Default. Try the same reset in the Default session, where it isn't offered, and you get a refusal.

拡張セッションではハードリセットが受理され、ECUはデフォルトへ再起動します。同じリセットを、それが提供されていないデフォルトセッションで試すと、拒否されます。

これは4行で表した1つの手順です ― セッションを開き、TesterPresentで維持し、作業し、最後にハードリセットで反映・後始末。ステップ1と4が、このページで学んだ2つのサービス $10 と $11 です ― もうそのすべてのバイトが読めます。

$11 = two bytes in (11 + reset type), two bytes back (51 + echo) before the ECU reboots. Choose the depth with the sub-function; remember the reboot returns the ECU to the Default Session and re-locks security. Refusals are 7F 11 <NRC>. You have now read two full UDS services end to end.

$11 ＝ 2バイト送信（11＋リセット種別）、再起動の前に 2バイト応答（51＋エコー）。深さはサブファンクションで選び、再起動でECUはデフォルトセッションへ戻りセキュリティも再施錠されることを忘れずに。拒否は 7F 11 <NRC>。これで2つのUDSサービスを端から端まで読めました。

For an ECU on a CAN bus, in a workshop, UDS is excellent: compact, fast, precise. Two-byte requests, deterministic timing, a fixed catalog of services. Everything you learned works and will keep working for years.

C27

SOVD: diagnostics for a connected, software-defined car

SOVD ― つながる、ソフトウェア定義の車のための診断
ISO 17978

SOVD — Service-Oriented Vehicle Diagnostics (ISO 17978) — is a diagnostic interface built the way modern software talks: a web API. Instead of hex bytes, a client asks for resources by human-readable name over HTTP, and gets structured JSON back. The same request works from a laptop in the workshop, a phone, or a cloud server across the internet.

SOVD ― Service-Oriented Vehicle Diagnostics（サービス指向車両診断、ISO 17978）― は、現代のソフトウェアの話し方でつくられた診断インターフェース、すなわち Web API です。16進のバイトの代わりに、クライアントは人が読める名前でリソースを HTTP 越しに要求し、構造化された JSON を受け取ります。同じ要求が、整備現場のノートPCからも、スマホからも、インターネット越しのクラウドサーバーからも動きます。

Why now? Cars became high-performance computers with apps, updates, and remote services. Diagnosing them purely with two-byte, onboard-only messages grew awkward. SOVD keeps UDS underneath where it fits, and adds a modern, web-shaped layer on top — remote-friendly, self-describing, and readable by people who are not UDS experts.

なぜ今か。車は、アプリ・更新・遠隔サービスを備えた高性能コンピューターになりました。それを2バイトの車載専用メッセージだけで診断するのは、次第に窮屈になりました。SOVDは、ふさわしい場所ではUDSを下敷きに残しつつ、その上に現代的でWeb的な層を足します ― 遠隔に強く、自己記述的で、UDSの専門家でなくても読めるものに。

Why the UDS foundation mattered · UDSの土台が効く理由 SOVD does not throw UDS away — for deep, safety-critical ECU access it still relies on UDS underneath. Knowing sessions, services, positive/negative responses, and the OSI split is exactly what lets you understand what SOVD adds and what it reuses. That is why we started here. SOVDはUDSを捨てません ― 深い安全重要なECUアクセスでは、下でUDSに頼り続けます。セッション、サービス、肯定/否定応答、OSIの分担を知っていることこそ、SOVDが「何を足し」「何を再利用するか」を理解する鍵です。だから、ここから始めたのです。

モジュール1は完了です。UDSのやり取りを読み、ECUとOSIスタックの中に位置づけ、$10 と $11 をバイト単位で扱えます。モジュール2 ― SOVDは、まさにここから始まります ― Web APIモデル、リソース、そしてそれが今学んだUDSの上にどう乗るのか。
